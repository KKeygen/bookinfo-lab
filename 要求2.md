实验目标
能够通过 kuboard 查看微服务系统总体信息，会使用 Prometheus 收集系统 metric，通过 Jaeger 查看系统 trace 和日志，通过 chaos-mash 注入故障，并观察系统变化。
背景知识
Kuboard —— 让 Kubernetes 管理变得可视化
官方网站：https://kuboard.cn/
[图片]
原生的 Kubernetes 主要依赖命令行工具 kubectl 进行管理，对于初学者而言学习曲线相当陡峭。Kuboard 是由国内开发者打造的一款轻量级图形化管理界面，它将复杂的 K8s 资源——如 Pod、Deployment、Service、Ingress 等——转化为直观的拓扑图与列表，让你无需记忆大量命令，就能清楚地看到集群中"正在发生什么"。
在本次实验中，你将主要通过 Kuboard 来完成以下操作：
- 观察资源状态：以可视化的方式浏览集群中各个微服务的部署情况与运行状态。
- 查看容器日志：直接在网页中查看容器的实时输出日志，方便排查问题。
- 监控节点资源：直观查看各节点的 CPU、内存分配情况，了解底层硬件资源池的使用状态。

---
Istio —— 掌管微服务间"交通"的服务网格
官方网站：https://istio.io/
[图片]
随着微服务数量的增加，服务与服务之间的通信管理会变得越来越复杂。服务网格（Service Mesh） 是专门为此而生的一层基础设施，而 Istio 是目前业界最广泛采用的服务网格实现。
Istio 的核心机制被称为 "边车模式（Sidecar Pattern）"。顾名思义，就像摩托车旁边的边车一样，Istio 会在每个业务微服务的 Pod 中自动注入一个代理容器（基于 Envoy 实现）。业务代码本身对此完全无感知——所有进出的网络请求，都会悄悄地先经过这个代理，再到达目的地。
这一设计带来了两个关键能力：
- 流量管理（Traffic Management）：这是本次实验的重点之一。通过修改 Istio 的配置，我们可以精确控制流量的走向，例如按权重将请求分配给同一服务的不同版本，从而实现灰度发布和 A/B 测试。实验中，你将亲手配置将 reviews 服务的流量按 20:40:40 的比例分发给其三个不同版本。
- 无侵入式监控：由于所有网络请求都要经过 Sidecar 代理，Istio 可以自动拦截并记录请求耗时、状态码等关键数据，并将其转发给 Jaeger 和 Prometheus 等下游系统进行分析——全程不需要修改任何业务代码。

Jaeger —— 在微服务迷宫中追踪一个请求的完整旅程
官方网站：https://www.jaegertracing.io/
[图片]
设想一下：一个用户点击页面后，请求可能依次经过了 A、B、C、D 四个微服务才得到响应。如果整个过程耗时 3 秒，你如何知道时间究竟"卡"在了哪里？这正是分布式追踪系统要解决的问题。
Jaeger 最初由 Uber 开源，现已成为 CNCF（云原生计算基金会）的毕业项目。它的工作原理是：当一个请求进入系统时，为其生成一个全局唯一的 Trace ID。无论该请求后续流经多少个微服务，都会携带这个 ID。最终，所有服务上报的片段信息会被汇聚在一起，还原出请求的完整调用链路。
在 Jaeger 的 UI 界面中，你可以看到一张瀑布流图：图中清晰地展示了微服务之间的调用情况，每一跳消耗了多少毫秒一目了然。如果某个服务发生阻塞，它在图中就会呈现为一条异常长的色条，帮助开发者实现快速的故障定位。

Prometheus —— 云原生时代的监控"大脑"
官方网站：https://prometheus.io/
[图片]
如果说 Jaeger 关注的是"一次请求经历了什么"，那么 Prometheus 关注的则是"整个系统随时间的变化趋势"。它是一个时间序列数据库与监控系统，继 Kubernetes 之后成为第二个从 CNCF 毕业的顶级项目，目前几乎是微服务监控领域的事实标准。
Prometheus 采用**"拉（Pull）"模式**进行数据采集——它不依赖各服务主动汇报，而是每隔固定时间（例如 15 秒）主动访问各个服务和节点暴露的 /metrics HTTP 接口，抓取当前的 CPU 使用率、内存占用、网络 IO 等指标数据。
Prometheus 还内置了一套强大的查询语言 PromQL。在本次实验中，你将体验如何编写 Python 代码，通过 PromQL 向 Prometheus 发送查询请求，精准地提取出 Bookinfo 应用各容器的相关指标。

Chaos Mesh —— 在受控环境中"主动制造故障"
官方网站：https://chaos-mesh.org/
[图片]
在传统的软件测试中，我们往往只关注系统在"一切正常"时的表现。然而在真实的生产环境里，网络抖动、节点过载、服务宕机等故障随时可能发生。一个真正健壮的系统，必须经得起这些意外的考验。混沌工程（Chaos Engineering） 正是为此而生的一种工程实践理念——它的核心思想是：与其等待故障在生产环境中猝不及防地发生，不如在受控的实验环境中主动、有计划地注入故障，提前发现系统的薄弱环节。
Chaos Mesh 是由 PingCAP（TiDB 的母公司）开源、目前已从 CNCF 毕业的云原生混沌工程平台。它与 Kubernetes 深度集成，允许开发者通过简单的配置，向运行中的集群精确地注入各种类型的故障，同时不影响集群的其他部分。
在本次实验中，你将主要体验以下两类故障注入：
- 网络故障（Network Chaos）：Chaos Mesh 可以模拟真实网络环境中的各种异常，例如在指定的微服务之间人为引入网络延迟或丢包。注入故障后，你可以观察 Jaeger 的调用链路图发生了怎样的变化——原本正常的请求耗时会在瀑布图中被"拉长"，直观地呈现出网络抖动对整个调用链的影响。
- CPU 负载（Stress Chaos）：Chaos Mesh 还可以对指定的容器施加 CPU 压力，模拟服务在高负载下的运行状态。注入后，你可以通过 Prometheus 查询该容器的资源指标，观察 CPU 使用率的异常攀升，体验监控系统在故障发现中所扮演的角色。

最后，我们通过 Python 脚本来获取集群状态，例如，下面这个脚本就能够获取 bookinfo 命名空间中每个服务的 CPU/内存占用和重启次数。
from __future__ import annotations

from dataclasses import dataclass
from typing import Any, Dict, List, Optional

import requests

@dataclass
class PrometheusConfig:
    base_url: str
    bearer_token: Optional[str] = None
    timeout: int = 15
    verify_ssl: bool = True

class PrometheusClient:
    def __init__(self, config: PrometheusConfig) -> None:
        self.config = config

    def _headers(self) -> Dict[str, str]:
        headers = {"Content-Type": "application/json"}
        if self.config.bearer_token:
            headers["Authorization"] = f"Bearer {self.config.bearer_token}"
        return headers

    def query(self, promql: str) -> List[Dict[str, Any]]:
        url = f"{self.config.base_url.rstrip('/')}/api/v1/query"
        resp = requests.get(
            url,
            headers=self._headers(),
            params={"query": promql},
            timeout=self.config.timeout,
            verify=self.config.verify_ssl,
        )
        resp.raise_for_status()

        payload = resp.json()
        if payload.get("status") != "success":
            raise RuntimeError(f"Prometheus query failed: {payload}")

        return payload["data"]["result"]

class BookinfoMonitor:
    def __init__(self, prometheus: PrometheusClient, namespace: str = "bookinfo") -> None:
        self.prometheus = prometheus
        self.namespace = namespace

    def get_pod_cpu_usage(self) -> List[Dict[str, Any]]:
        """
        查询每个 Pod 的 CPU 使用量（单位：cores）
        """
        promql = f'''
        sum by (pod) (
          rate(container_cpu_usage_seconds_total{{
            namespace="{self.namespace}",
            container!="",
            container!="POD",
            image!=""
          }}[5m])
        )
        '''
        results = self.prometheus.query(promql)

        data = []
        for item in results:
            data.append({
                "pod": item["metric"].get("pod", "unknown"),
                "cpu_cores": float(item["value"][1]),
            })
        return sorted(data, key=lambda x: x["pod"])

    def get_pod_memory_usage(self) -> List[Dict[str, Any]]:
        """
        查询每个 Pod 的内存使用量（单位：bytes）
        """
        promql = f'''
        sum by (pod) (
          container_memory_working_set_bytes{{
            namespace="{self.namespace}",
            container!="",
            container!="POD",
            image!=""
          }}
        )
        '''
        results = self.prometheus.query(promql)

        data = []
        for item in results:
            data.append({
                "pod": item["metric"].get("pod", "unknown"),
                "memory_bytes": float(item["value"][1]),
            })
        return sorted(data, key=lambda x: x["pod"])

    def get_pod_restart_count(self) -> List[Dict[str, Any]]:
        """
        查询每个 Pod 的重启次数
        """
        promql = f'''
        sum by (pod) (
          kube_pod_container_status_restarts_total{{
            namespace="{self.namespace}"
          }}
        )
        '''
        results = self.prometheus.query(promql)

        data = []
        for item in results:
            data.append({
                "pod": item["metric"].get("pod", "unknown"),
                "restart_count": int(float(item["value"][1])),
            })
        return sorted(data, key=lambda x: x["pod"])

def bytes_to_mib(num_bytes: float) -> float:
    return num_bytes / 1024 / 1024

        )
    )

    monitor = BookinfoMonitor(client, namespace="bookinfo")

    print("=== Pod CPU Usage ===")
    for item in monitor.get_pod_cpu_usage():
        print(f"{item['pod']}: {item['cpu_cores']:.4f} cores")

    print("\n=== Pod Memory Usage ===")
    for item in monitor.get_pod_memory_usage():
        print(f"{item['pod']}: {bytes_to_mib(item['memory_bytes']):.2f} MiB")

    print("\n=== Pod Restart Count ===")
    for item in monitor.get_pod_restart_count():
        print(f"{item['pod']}: {item['restart_count']}")

其本质是 Prometheus 提供了查询的 API，python 客户端负责构造对应的 Prometheus 拆查询语句，向 Prometheus 的 API 发送请求，得到 Prometheus 返回的集群信息。

网络故障
可以通过以下配置切断 productpage 和 reviews 微服务之间的通信。
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: partition-productpage-reviews
  namespace: bookinfo 微服务所在的命名空间
spec:
  action: partition               # 完全切断，双向
  mode: all
  selector:
    namespaces: [bookinfo 微服务所在的命名空间]
    labelSelectors:
      app: productpage
  direction: both
  target:
    mode: all
    selector:
      namespaces: [bookinfo 微服务所在的命名空间]
      labelSelectors:
        app: reviews
  duration: "5m"

CPU 负载压力
可以通过如下配置为 prodcutpage 配置 CPU 负载：
apiVersion: chaos-mesh.org/v1alpha1
kind: StressChaos
metadata:
  name: cpu-stress-productpage
  namespace: bookinfo 微服务所在的命名空间
spec:
  mode: one                        # 随机选一个 Pod
  selector:
    namespaces: [bookinfo 微服务所在的命名空间]
    labelSelectors:
      app: productpage
  stressors:
    cpu:
      workers: 4                   # 4 个 goroutine 跑满 CPU
      load: 100                    # 百分比
  duration: "2m"
保存文件之后应用该配置：
> kubectl apply -f cpu-stress.yaml
访问 kuboard ，可以发现 productpage 这个 node CPU 占用率飙升。
[图片]
通过以下命令停止故障注入。
> kubectl delete -f cpu-stress.yaml
停止故障注入之后 CPU 占用率回归正常。
